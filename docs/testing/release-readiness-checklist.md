# 发布前最后巡检与真机测试清单

适用目标：体验版/小范围灰度发布，验证“是否有人愿意发布和申请加入周边行程”。

## 当前自动化结果

已通过：

- `npm test`：13/13 通过，覆盖首页、登录页、发布页、发布成功页的轻量逻辑和 UI 结构。
- `npm run test:e2e`：通过，真实点击首页筛选、未登录跳登录、我的页入口、登录页、发布页、发布成功页。
- `npm run test:core`：通过，真实点击核心业务链路：发布、申请、同意、移除、退出、双方行程通知展示。

自动化未覆盖或只使用 mock：

- 微信手机号授权系统弹窗。
- 头像选择系统弹窗。
- 真实云数据库写入、权限、索引。
- 两个真实微信号之间的真实 openid 数据隔离。
- 真机视觉兼容、键盘遮挡、网络弱环境。

## 发布前必须确认

### 1. 云环境与集合

在云开发控制台确认以下集合存在：

- `users`
- `trips`
- `applies`
- `quick_attractions`
- `user_stats`
- `feedbacks`
- `user_spots`
- `place_view_stats`
- `banners`
- `places`
- `wants`
- `messages`
- `comments`

重点：当前首页/发布/详情核心链路主要依赖 `quick_attractions`，不是只依赖 `places`。

### 2. 云函数

确认已部署：

- `login`
- `api`

`cloudbaserc.json` 里没有列 `init-data` 函数，如果你需要用云函数初始化数据，要确认它已单独部署或手动导入过数据。

### 3. 初始数据

确认 `quick_attractions` 至少有 5-10 条真实可展示地点，并且字段包含：

- `_id`
- `name`
- `coverImage` 或 `image`
- `location` 或 `address`

### 4. 未完成入口风险

发布体验版前建议隐藏或避免引导用户点击：

- `pages/chat/chat`：TODO 空页。
- `pages/comments/comments`：TODO 空页。
- `pages/followers/followers`：TODO 空页。
- `pages/search/search`：TODO 空页。
- `pages/messages/messages`：使用 mock 消息数据。
- `profile` 的“我的收藏”：点击会提示“功能开发中”。

如果不想现在改代码，灰度测试话术里不要引导用户使用这些入口。

### 5. 假数据与兜底 mock

注意以下页面有 mock/兜底逻辑：

- `profile` 统计目前是固定模拟数据。
- `trip-detail` 云端加载失败时会展示模拟行程。
- `place-detail` 云端加载失败时会展示模拟地点。
- `index` 云端加载失败时会展示模拟地点。

真机测试时要特别确认：断网或云函数失败时是否会误导用户看到假数据。

## 真机测试准备

准备两台手机或两个微信账号：

- A：发起人。
- B：申请人。

准备环境：

- 使用体验版或开发版。
- 两个账号都清理本地登录状态后重新登录。
- 云开发控制台打开 `users`、`trips`、`applies` 集合，方便对照数据。

## 15 分钟真机测试主流程

### A. 登录

1. A 打开小程序。
2. 进入“我的”，点击登录。
3. 不勾选协议，点击微信一键登录。
4. 预期：提示“请先同意用户协议”。
5. 勾选协议，再点击微信一键登录。
6. 允许手机号授权。
7. 设置头像和昵称，完成登录。
8. 预期：回到行程首页或首页，云端 `users` 有 A 用户。

对 B 重复以上登录流程。

### B. 发布行程

1. A 进入行程首页。
2. 点击发布按钮。
3. 直接点击发布。
4. 预期：提示必填项。
5. 依次填写目的地、出发地、日期、手机号。
6. 点击发布。
7. 预期：进入发布成功页。
8. 云端确认：`trips` 新增记录，`creatorId` 是 A 的 openid，`participants` 里包含 A。

### C. B 申请加入

1. B 打开 A 发布的行程详情。
2. 点击“申请加入”。
3. 不填手机号提交一次。
4. 预期：提示“请填写联系方式”。
5. 填错手机号提交一次。
6. 预期：提示“请输入正确的手机号”。
7. 填正确手机号和备注，提交申请。
8. 预期：提示“申请已发送”。
9. B 进入行程通知。
10. 预期：看到“申请中”。
11. 云端确认：`applies` 至少新增两条或一组申请记录，A/B 双方都能看到对应通知。

### D. A 同意申请

1. A 进入“我的” -> “行程通知”。
2. 找到 B 的申请。
3. 点击“同意”。
4. 预期：提示“操作成功”，通知变成“已同意”。
5. B 进入行程通知。
6. 预期：看到“已同意”。
7. B 打开行程详情。
8. 预期：底部显示“退出行程”，发起人联系方式可见。
9. 云端确认：`trips.participants` 包含 B，`currentCount` 增加，`needCount` 减少。

### E. A 移除 B

1. A 打开行程详情。
2. 点击 B 的成员头像。
3. 点击“移除成员”。
4. 点击“确认移除”。
5. 预期：提示“已移除成员”，成员列表不再显示 B。
6. B 进入行程通知。
7. 预期：看到“已移除”通知。
8. 云端确认：`trips.participants` 不再包含 B。

### F. B 退出行程

用新行程重复 B 申请、A 同意后：

1. B 打开已加入的行程详情。
2. 点击“退出行程”。
3. 确认退出。
4. 预期：提示“已退出行程”，返回上一页。
5. A 进入行程通知。
6. 预期：看到“B 退出了您的行程”或“已退出”通知。
7. 云端确认：`trips.participants` 不再包含 B，名额恢复。

## 兼容性真机检查

至少检查：

- iPhone 小屏：发布页底部按钮是否遮挡输入框。
- 安卓常见机型：登录页协议、手机号按钮是否可点。
- 键盘弹起：手机号输入、备注输入是否被遮挡。
- 弱网：发布、申请、同意时 loading 是否正常，失败提示是否清楚。
- 分享：A 分享行程给 B，B 打开后是否进入正确行程详情。

## 灰度发布判断指标

第一批 20-50 人测试，建议看：

- 访问人数。
- 登录人数。
- 发布行程人数。
- 申请加入次数。
- 发布后平均申请数。
- 申请通过数。
- 真实加微信/成行意向数。
- 反馈中出现最多的问题。

建议放行体验版灰度的最低标准：

- 核心流程无阻塞 bug。
- 至少 3 个非熟人用户愿意发布或申请。
- 至少 1-2 个真实出行沟通意向。
- 反馈主要是“想要更多行程/地点/人”，而不是“登录不了/看不到/不知道怎么用”。
