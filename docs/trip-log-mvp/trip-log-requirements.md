# 行程日志 MVP 需求说明

本文档是当前确认版需求，用于指导行程详情页中的“旅途记录 / 行程日志”功能开发与验收。

## 一、功能定位

行程日志是挂载在单个行程详情页上的旅行时间线功能。

核心目标：

- 让别人看到行程正在真实发生。
- 让行程过程中的文字和图片动态沉淀下来。
- 行程结束后，保留一份所有人都可以查看的公开记录。

第一版定位是“行程时间线”，不是聊天室，也不是开放社区动态。

## 二、可见性规则

- 所有用户都可以查看已发布的旅途记录。
- 旅途记录只属于当前行程，不进入全局动态流。
- 如果当前行程没有发布任何日志，则不展示旅途记录模块。
- 行程结束后，已发布日志继续公开展示。

## 三、行程状态模型

行程使用两个核心字段表达状态：

```js
tripStage: 'not_started' | 'ongoing' | 'ended' | 'cancelled'
status: 'open' | 'stopped'
```

字段职责：

- `tripStage` 表示行程生命周期。
- `status` 表示招募开关。
- `needCount` 用于动态计算是否即将满员或已满员。

状态展示优先级：

```js
if (tripStage === 'cancelled') return '已取消'
if (tripStage === 'ongoing') return '进行中'
if (tripStage === 'ended') return '已结束'
if (status === 'stopped') return '停止招募'
if (needCount <= 0) return '已满员'
if (needCount === 1) return '即将满员'
return '招募中'
```

说明：

- `full` 和 `almost_full` 不写入数据库。
- `已满员` 由 `needCount <= 0` 计算。
- `即将满员` 由 `needCount === 1` 计算。
- `cancelled` 不写入 `status`，只写入 `tripStage`。

## 四、首页状态筛选

首页状态筛选包含：

- 全部
- 招募中
- 即将满员
- 已满员
- 进行中
- 已结束

筛选规则：

- 全部：展示招募中、即将满员、已满员、进行中。
- 招募中：展示招募中、即将满员。
- 即将满员：只展示 `tripStage === 'not_started' && status === 'open' && needCount === 1`。
- 已满员：只展示 `tripStage === 'not_started' && needCount <= 0`。
- 进行中：只展示 `tripStage === 'ongoing'`。
- 已结束：只展示 `tripStage === 'ended'`。

## 五、我的行程页状态规则

我的行程页保留身份维度 tab：

- 全部
- 我发起的
- 我参与的
- 已结束

我的行程页不要求改成首页同款状态筛选结构。

但我的行程页中的状态展示必须和首页、详情页保持一致：

- 使用同一套状态优先级。
- `tripStage` 优先于 `status` 和 `needCount`。
- 已满员、即将满员只由 `needCount` 动态计算。

后续如果维护成本变高，可以抽公共方法，例如：

```js
getTripDisplayStatus(trip)
```

## 六、开始与结束规则

旅途记录必须由发起人手动开始、手动结束。

开始规则：

- 只有行程发起人可以点击 `开始出发`。
- `开始出发` 按钮仅在行程当天展示。
- 点击开始后，`tripStage` 更新为 `ongoing`。
- 点击开始后，旅途记录发布通道开启。
- 开始后，成员不能退出行程。
- 开始后，发起人不能移除成员。

结束规则：

- 只有行程发起人可以点击 `结束行程`。
- 点击结束前必须二次确认。
- 确认提示需要表达：结束后不能继续发布旅途记录，将保存现有记录。
- 点击结束后，`tripStage` 更新为 `ended`。
- 点击结束后，旅途记录发布通道关闭。
- 已结束后，只能查看旅途记录，不能继续发布。

系统不自动开始行程，也不自动结束行程。

## 七、发布权限规则

默认发布权限：

- 发起人默认拥有发布权限。
- 其他成员默认没有发布权限。

授权规则：

- 只有发起人可以授权成员发布旅途记录。
- 只有已加入当前行程的成员可以被授权。
- 授权只对当前行程生效。
- 只有 `tripStage === 'ongoing'` 时才允许授权或取消授权。
- 被授权成员只能发布日志，不能删除日志，不能管理行程。

授权人数上限：

- 授权人数上限来自数据库配置或行程快照。
- 当前默认值为 `2`。
- 代码中不能直接把 `2` 作为唯一业务来源。

推荐配置：

```js
{
  _id: 'trip_log',
  logPublisherLimit: 2,
  logMaxCount: 15,
  updatedAt: Date.now()
}
```

创建行程时，将配置快照写入行程：

```js
{
  logPublisherLimit: 2,
  logMaxCount: 15
}
```

## 八、日志内容规则

每条日志支持：

- 文字
- 图片

限制规则：

- 每条日志最多 3 张图片。
- 文字最多 200 字。
- 文字和图片不能同时为空。
- 第一版不支持编辑日志。
- 第一版不支持评论、点赞、浏览记录、定位打点。

## 九、日志数量上限

单个行程最多发布 `logMaxCount` 条日志。

当前默认值：

```js
logMaxCount: 15
```

规则：

- `logMaxCount` 从数据库配置或行程快照读取。
- 发布成功后，`logCount` 增加 1。
- 删除日志后，`logCount` 必须递减。
- 重复删除同一条日志不能重复递减 `logCount`。

并发说明：

- MVP 阶段允许暂缓处理极端并发问题。
- 当前发布者数量有限，风险可接受。
- 用户量扩大后，可通过原子更新或更严格的并发控制处理。

## 十、删除规则

- 只有行程发起人可以删除任意日志。
- 被授权成员不能删除日志，包括自己发布的日志。
- 其他成员和游客不能删除日志。
- 删除后前端不再展示该日志。
- 删除后 `logCount` 递减。

## 十一、旅途记录展示规则

排序：

- 按发布时间正序展示。
- 越早发布的日志排在上方。
- 越晚发布的日志排在下方。

每条日志展示：

- 发布者头像
- 发布者昵称
- 发布时间
- 文字内容
- 1 到 3 张图片

时间展示：

- 当天发布显示 `HH:mm`。
- 跨天发布显示 `MM-DD HH:mm`。

## 十二、UI 规则

详情页头部：

- 沿用当前行程详情页头部 UI。
- 不做大范围视觉改版。

未开始状态：

- 除了发起人可看到 `开始出发` 按钮，不额外增加新模块。
- 不展示旅途记录模块。
- 不展示旅途记录提示。

进行中且无日志：

- 不展示日志列表模块。
- 发起人展示 `发布日志` 和 `结束行程`。
- 被授权成员展示 `发布日志`。
- 无权限成员展示 `进行中`。
- 页面展示提示，让用户知道旅途记录已开启。

有日志后：

- 页面主体采用两个 tab：`行程` 和 `旅途记录`。
- `行程` tab 使用当前详情页已有 UI。
- `旅途记录` tab 使用已确认的 B 方案日记流样式。
- `旅途记录` 标题区在列表外层。
- 每条动态使用独立白色卡片。
- 图片横向排版，1 张、2 张、3 张都尽量铺满当前行宽。
- 绿色作为 tab 高亮、按钮和标签辅助色。

已结束状态：

- 底部按钮展示 `已结束`。
- 不展示发布入口。
- 如果有日志，正常展示旅途记录。
- 如果没有日志，不展示旅途记录模块。

## 十三、成员关系锁定

当 `tripStage === 'ongoing'` 时：

- 成员不能退出行程。
- 发起人不能移除成员。

如果用户尝试操作，需要提示：

- 行程已开始。
- 开始后不可退出或移除成员。

## 十四、第一版不做

MVP 第一版不做：

- 评论
- 点赞
- 谁看过
- 访客记录
- 定位打点
- 日志编辑
- 复杂权限组
- 全局动态流
- 系统自动开始
- 系统自动结束
