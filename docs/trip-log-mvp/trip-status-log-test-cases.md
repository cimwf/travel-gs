# 行程状态与旅途记录测试用例

本文档用于验证“行程状态模型修复”和“旅途记录功能调整”是否符合当前产品规则。建议在代码修改完成后，按照本文档逐项测试，并在最后生成测试报告。

## 一、测试范围

本次测试覆盖：

- 行程生命周期 `tripStage`
- 招募状态 `status`
- 首页状态展示与筛选
- 我的行程页状态展示
- 行程详情页开始/结束流程
- 旅途记录发布权限
- 授权发布人规则
- 日志数量上限
- 删除日志后 `logCount` 递减
- 系统自动处理逻辑是否已删除

## 二、基础字段期望

### 1. trips 集合核心字段

测试前确认行程数据应包含：

```js
{
  tripStage: 'not_started',
  status: 'open',
  needCount: 2,

  logStartedAt: null,
  logEndedAt: null,

  logPublisherLimit: 2,
  logAuthorizedPublisherIds: [],

  logMaxCount: 15,
  logCount: 0,
  lastLogAt: null
}
```

### 2. 不应继续使用的字段

以下字段不应再作为业务判断依据：

```js
logStatus
logAutoEndDays
logAutoEndAt
```

说明：

- 如果旧数据中仍存在这些字段，前端和云函数也不应依赖它们。
- 新创建行程不应再写入 `logStatus`。
- 系统不应再自动结束行程或旅途记录。

## 三、行程状态展示测试

### TC-001 未开始且可招募

前置数据：

```js
tripStage = 'not_started'
status = 'open'
needCount = 2
```

预期结果：

- 首页展示：招募中
- 我的行程页展示：招募中
- 详情页展示：招募中
- 可以报名或申请加入

### TC-002 未开始且即将满员

前置数据：

```js
tripStage = 'not_started'
status = 'open'
needCount = 1
```

预期结果：

- 首页展示：即将满员
- 我的行程页展示：即将满员
- 详情页展示：即将满员
- 该状态不应写入数据库，只能由 `needCount` 计算得出

### TC-003 未开始且已满员

前置数据：

```js
tripStage = 'not_started'
status = 'open'
needCount = 0
```

预期结果：

- 首页展示：已满员
- 我的行程页展示：已满员
- 详情页展示：已满员
- 不能继续报名
- 该状态不应写入数据库，只能由 `needCount` 计算得出

### TC-004 未开始且停止招募

前置数据：

```js
tripStage = 'not_started'
status = 'stopped'
needCount = 2
```

预期结果：

- 首页展示：停止招募
- 我的行程页展示：停止招募
- 详情页展示：停止招募
- 不能继续报名

### TC-005 进行中

前置数据：

```js
tripStage = 'ongoing'
status = 'open'
needCount = 2
```

预期结果：

- 首页展示：进行中
- 我的行程页展示：进行中
- 详情页展示：进行中
- 状态展示不再受 `needCount` 影响
- 不允许退出行程
- 不允许移除成员
- 允许有权限的人发布旅途记录

### TC-006 已结束

前置数据：

```js
tripStage = 'ended'
status = 'open'
needCount = 2
```

预期结果：

- 首页展示：已结束
- 我的行程页展示：已结束
- 详情页展示：已结束
- 状态展示不再受 `needCount` 影响
- 不允许发布旅途记录
- 只能查看已发布旅途记录

### TC-007 已取消

前置数据：

```js
tripStage = 'cancelled'
status = 'open'
needCount = 2
```

预期结果：

- 首页展示：已取消
- 我的行程页展示：已取消
- 详情页展示：已取消
- 不允许报名
- 不允许开始行程
- 不允许发布旅途记录

## 四、状态筛选测试

### TC-008 首页“全部”筛选

准备数据：

- 行程 A：招募中
- 行程 B：即将满员
- 行程 C：已满员
- 行程 D：进行中
- 行程 E：已结束
- 行程 F：已取消

预期结果：

- “全部”展示 A、B、C、D
- “全部”不展示 E、F

### TC-009 首页“招募中”筛选

准备数据：

- 行程 A：招募中
- 行程 B：即将满员
- 行程 C：已满员

预期结果：

- “招募中”展示 A、B
- “招募中”不展示 C

### TC-010 首页其他筛选

预期结果：

- “即将满员”只展示 `needCount === 1` 的未开始行程
- “已满员”只展示 `needCount <= 0` 的未开始行程
- “进行中”只展示 `tripStage === 'ongoing'` 的行程
- “已结束”只展示 `tripStage === 'ended'` 的行程

### TC-011 我的行程页状态展示一致性

测试步骤：

- 使用与首页相同的数据。
- 进入我的行程页。
- 分别查看全部、我发起的、我参与的、已结束 tab 中的行程卡片。

预期结果：

- 我的行程页保留身份维度 tab：全部、我发起的、我参与的、已结束。
- 我的行程页不要求改成首页同款状态筛选结构。
- 我的行程页与首页使用同一套状态计算规则，或至少保持完全一致的状态优先级。
- 相同状态在两个页面展示文案一致。
- `tripStage` 优先于 `status` 和 `needCount`。
- `已满员` 与 `即将满员` 仍然只由 `needCount` 动态计算，不写入数据库状态。

## 五、开始行程测试

### TC-012 发起人手动开始行程

前置条件：

- 当前用户是行程发起人。
- `tripStage = 'not_started'`。

测试步骤：

- 进入行程详情页。
- 点击“开始出发”。
- 确认开始。

预期结果：

- `tripStage` 更新为 `ongoing`。
- `logStartedAt` 写入当前时间戳。
- 页面按钮变为“结束行程”。
- 页面展示旅途记录功能提示。
- 未发布任何日志时，不展示日志列表模块。

### TC-013 非发起人不能开始行程

前置条件：

- 当前用户不是行程发起人。
- `tripStage = 'not_started'`。

测试步骤：

- 进入行程详情页。

预期结果：

- 不展示“开始出发”按钮。
- 如果调用云函数强行开始，应返回无权限错误。

### TC-014 已结束行程不能重新开始

前置条件：

```js
tripStage = 'ended'
```

测试步骤：

- 调用开始行程接口。

预期结果：

- 接口返回失败。
- `tripStage` 仍然是 `ended`。

## 六、结束行程测试

### TC-015 发起人手动结束行程

前置条件：

- 当前用户是行程发起人。
- `tripStage = 'ongoing'`。

测试步骤：

- 进入行程详情页。
- 点击“结束行程”。
- 弹窗确认。

预期结果：

- 弹窗提示：结束后不能继续发布旅途记录，将保存现有记录。
- 确认后 `tripStage` 更新为 `ended`。
- `logEndedAt` 写入当前时间戳。
- 发布日志入口关闭。
- 按钮展示为“已结束”。
- 已发布日志继续展示。

### TC-016 非发起人不能结束行程

前置条件：

- 当前用户不是行程发起人。
- `tripStage = 'ongoing'`。

测试步骤：

- 调用结束行程接口。

预期结果：

- 接口返回无权限错误。
- `tripStage` 仍然是 `ongoing`。

## 七、旅途记录发布测试

### TC-017 发起人发布旅途记录

前置条件：

- 当前用户是行程发起人。
- `tripStage = 'ongoing'`。
- `logCount < logMaxCount`。

测试步骤：

- 点击发布日志。
- 输入文字。
- 上传 1 到 3 张图片。
- 提交。

预期结果：

- 发布成功。
- 新日志按正序展示在时间线中。
- `logCount` 增加 1。
- `lastLogAt` 更新为当前发布时间。

### TC-018 未开始不能发布旅途记录

前置条件：

```js
tripStage = 'not_started'
```

测试步骤：

- 尝试调用发布日志接口。

预期结果：

- 发布失败。
- 提示行程尚未开始，不能发布旅途记录。
- `logCount` 不变化。

### TC-019 已结束不能发布旅途记录

前置条件：

```js
tripStage = 'ended'
```

测试步骤：

- 尝试调用发布日志接口。

预期结果：

- 发布失败。
- 提示行程已结束，不能发布旅途记录。
- `logCount` 不变化。

### TC-020 未授权成员不能发布旅途记录

前置条件：

- 当前用户是行程成员。
- 当前用户不是发起人。
- 当前用户不在 `logAuthorizedPublisherIds` 中。
- `tripStage = 'ongoing'`。

测试步骤：

- 尝试发布日志。

预期结果：

- 发布失败。
- 提示没有发布权限。
- `logCount` 不变化。

### TC-021 被授权成员可以发布旅途记录

前置条件：

- 当前用户是行程成员。
- 当前用户在 `logAuthorizedPublisherIds` 中。
- `tripStage = 'ongoing'`。

测试步骤：

- 发布一条旅途记录。

预期结果：

- 发布成功。
- `logCount` 增加 1。
- 日志展示当前用户头像、昵称、发布时间。

### TC-022 每条日志最多 3 张图

测试步骤：

- 尝试发布 4 张图片。

预期结果：

- 前端阻止选择或提交。
- 云函数也必须校验失败。
- 不创建日志。
- `logCount` 不变化。

### TC-023 日志文字长度限制

测试步骤：

- 输入超过限制的文字。
- 提交日志。

预期结果：

- 前端提示超出字数限制。
- 云函数也必须校验失败。
- 不创建日志。
- `logCount` 不变化。

## 八、授权发布人测试

### TC-024 发起人授权成员

前置条件：

- 当前用户是发起人。
- 被授权用户是当前行程成员。
- 当前授权人数小于 `logPublisherLimit`。

测试步骤：

- 发起人授权一个成员发布旅途记录。

预期结果：

- 授权成功。
- 被授权用户 ID 写入 `logAuthorizedPublisherIds`。
- 被授权用户可以发布旅途记录。

### TC-025 授权人数不能超过数据库配置

前置条件：

```js
logPublisherLimit = 2
logAuthorizedPublisherIds.length = 2
```

测试步骤：

- 发起人继续授权第 3 个成员。

预期结果：

- 授权失败。
- 提示授权人数已达上限。
- 上限必须来自数据库字段，不能是代码硬编码。

### TC-026 非成员不能被授权

前置条件：

- 被授权用户不是当前行程成员。

测试步骤：

- 发起人尝试授权该用户。

预期结果：

- 授权失败。
- 提示只能授权当前行程成员。

### TC-027 非发起人不能授权

前置条件：

- 当前用户不是行程发起人。

测试步骤：

- 尝试调用授权接口。

预期结果：

- 接口返回无权限错误。
- `logAuthorizedPublisherIds` 不变化。

## 九、日志数量上限测试

### TC-028 达到日志数量上限后不能继续发布

前置条件：

```js
logMaxCount = 15
logCount = 15
tripStage = 'ongoing'
```

测试步骤：

- 发起人或被授权成员尝试继续发布日志。

预期结果：

- 发布失败。
- 提示本次行程的旅途记录已达到上限。
- `logMaxCount` 必须来自数据库字段，不能是代码硬编码。

### TC-029 未达到上限可以发布

前置条件：

```js
logMaxCount = 15
logCount = 14
tripStage = 'ongoing'
```

测试步骤：

- 发布一条日志。

预期结果：

- 发布成功。
- `logCount` 更新为 15。
- 再次发布时失败。

## 十、删除日志测试

### TC-030 发起人可以删除任意日志

前置条件：

- 当前用户是发起人。
- 当前行程已有日志。
- `logCount = 3`。

测试步骤：

- 删除其中一条日志。

预期结果：

- 删除成功。
- 页面不再展示该日志。
- `logCount` 更新为 2。

### TC-031 被授权成员不能删除日志

前置条件：

- 当前用户是被授权成员。
- 当前用户发布过日志。

测试步骤：

- 尝试删除自己发布的日志。

预期结果：

- 删除失败。
- 提示只有发起人可以删除旅途记录。
- `logCount` 不变化。

### TC-032 删除后可以继续发布到上限

前置条件：

```js
logMaxCount = 15
logCount = 15
tripStage = 'ongoing'
```

测试步骤：

- 发起人删除 1 条日志。
- 再发布 1 条新日志。

预期结果：

- 删除后 `logCount = 14`。
- 新日志发布成功。
- 发布后 `logCount = 15`。

## 十二、自动处理删除验证

### TC-035 系统不自动开始行程

前置条件：

- 当前日期已到行程当天。
- `tripStage = 'not_started'`。

测试步骤：

- 打开首页。
- 打开详情页。
- 调用行程详情接口。

预期结果：

- `tripStage` 仍然是 `not_started`。
- 页面不应自动变成进行中。
- 只有发起人点击“开始出发”后才会变成进行中。

### TC-036 系统不自动结束行程

前置条件：

- 行程已开始。
- `tripStage = 'ongoing'`。
- 已经超过原先自动结束时间。

测试步骤：

- 打开首页。
- 打开详情页。
- 查询旅途记录列表。
- 尝试发布一条日志。

预期结果：

- `tripStage` 仍然是 `ongoing`。
- 系统不自动改为 `ended`。
- 仍然允许有权限的人发布日志。
- 只有发起人点击“结束行程”后才会变成已结束。

## 十三、旅途记录展示测试

### TC-037 没有日志时不展示日志模块

前置条件：

```js
tripStage = 'ongoing'
logCount = 0
```

测试步骤：

- 打开行程详情页。
- 切换到旅途记录 tab。

预期结果：

- 不展示日志列表模块。
- 发起人展示“发布日志”和“结束行程”按钮。
- 被授权成员展示“发布日志”入口。
- 无权限成员只展示进行中状态或相关提示。

### TC-038 有日志后展示时间线

前置条件：

```js
logCount > 0
```

测试步骤：

- 打开旅途记录 tab。

预期结果：

- 展示旅途记录标题。
- 每条日志以独立卡片展示。
- 按发布时间正序排列。
- 展示发布者头像、昵称、发布时间、文字、图片。

### TC-039 图片数量展示

测试步骤：

- 分别发布 1 张图、2 张图、3 张图的日志。

预期结果：

- 1 张图横向铺满可用宽度。
- 2 张图横向均分宽度。
- 3 张图横向均分宽度。
- 图片排版不使用复杂花哨布局。

## 十四、建议测试顺序

建议按以下顺序测试：

1. 数据库字段是否正确。
2. 首页状态展示。
3. 首页状态筛选。
4. 我的行程页状态展示与筛选。
5. 发起人开始行程。
6. 发起人发布日志。
7. 授权成员发布日志。
8. 日志数量上限。
9. 删除日志与 `logCount` 递减。
10. 发起人结束行程。
11. 已结束后禁止发布。
12. 验证系统不会自动开始或自动结束。
