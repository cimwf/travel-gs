先做“同城约伴”，再长成“周边活动社区”

你的三个痛点不是三条完全分散的产品线。景点榜单解决“去哪儿”，行程和活动解决“和谁去、一起做什么”。当前最适合先把第二点做扎实，然后把第三点并入同一套发布系统：发布的不是单纯行程，而是“活动”。旅行、吃饭、露营、游戏都只是活动类型。

推荐名称 近伴

“附近的人，一起出发”。名字短、好记，不把产品锁死在旅游里，后续扩展吃饭、露营、桌游、城市活动都自然。

功能 1

景点月榜

用户选择月份，看到北京及周边当月热门景点。点赞越多排序越靠前。它更像内容入口和灵感入口，适合第二阶段做。

功能 2

同行招募

这是当前 MVP 核心。用户发布行程，附近或同城用户报名/加入，形成真实线下出行关系。先把这条链路做顺。

功能 3

活动发布

建议和行程页合并。把“行程”升级成“活动”，字段加活动类型即可：旅行、吃饭、露营、游戏、徒步、摄影。

命名备选

优先选不太窄、容易口口相传、能覆盖旅行和本地活动的名字。

近伴 推荐。强调附近、陪伴、一起出发，适合从旅行扩展到饭局、露营、游戏。
周末搭子 更年轻、更社交，但稍微偏口语，适合轻松活动社区。
北边走走 有北京周边气质，适合旅行感，但后续扩展到吃饭游戏会稍窄。
同去 极简，表达明确，但需要更强视觉识别来撑起品牌记忆。

推荐规划

不要一开始同时做太多入口。先验证“发布 - 报名 - 成行 - 评价/沉淀”的闭环。

Phase 1 活动招募 MVP 保留当前行程页，但数据模型改成活动：类型、地点、时间、人数、费用、集合地、说明、报名状态。
Phase 2 活动类型扩展 旅行之外增加吃饭、露营、游戏、徒步。列表筛选改成活动类型 + 时间 + 地点。
Phase 3 景点月榜 上线月份榜单、点赞排序、景点详情。景点详情页里可以直接发起“去这里”的活动。
Phase 4 信任与留存 加入报名审核、活动通知、用户主页、参加记录、活动评价，让陌生人线下同行更安心。

三个人物与 UI 方向

三个人物分别代表户外旅行者、饭局/活动组织者、露营游戏参与者。UI 图里把景点月榜、同行招募、发布活动放在同一个产品体系里。

页面结构建议

  • 首页：活动列表，默认展示本周末/附近/热门。
  • 发布：统一叫“发布活动”，活动类型包含旅行、吃饭、露营、游戏。
  • 景点：第二阶段做“月度热门景点”，景点可点赞，也可一键发起活动。
  • 我的：我的活动、报名通知、资料认证、历史参与。
  • 登录：延续当前蓝白轻量风格，不需要大改。
