一套围绕“月份景点”和“可配置活动”的完整 UI 方案

这版把景点页从普通推荐改成按月份浏览：每个月的自然风景、花期、避暑、红叶、雪景都不同。活动发布也不再固定旅行、吃饭、露营、游戏四类，而是从后台配置活动类型，用户端自动展示。

景点逻辑 月份是主筛选，点赞数决定当月排序，景点详情可直接发起“去这里”的活动。
活动逻辑 统一叫活动，旅行只是活动类型。字段由类型决定，后台可新增和启停。
MVP 顺序 先做活动闭环，再接景点月榜，最后做后台运营配置和活动质量治理。

推荐产品路径

01 首页默认展示“本月值得去”和“本周可加入活动”，先给用户一个明确行动入口。
02 景点月份切换 + 当月榜单 + 点赞排序，解决“这个月份去哪儿玩”的问题。
03 活动按类型、时间、地点筛选，用户可以加入别人发的活动，也可以自己发布。
04 发布活动类型来自后台配置，新增类型后用户端无需重新设计入口。
05 我的我的发布、我的报名、通知、资料信任记录，保证线下同行更安心。

这组界面覆盖小程序第一版可上线的关键流程：按月份发现景点、浏览活动、动态类型发布、活动详情、我的页面。

设计落地要点

月份景点模型景点不要只有全局热度，建议有 monthStats：month、likeCount、reasonTags、seasonText。用户切换月份时读取对应榜单。
活动类型模型类型配置建议包含 name、icon、sort、enabled、coverPreset、requiredFields、optionalFields。发布页完全读取配置。
命名与视觉小程序名用“行”，页面里用“同行”“想去”“发起活动”这些动作词，让一个字品牌更容易被理解。
